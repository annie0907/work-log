## 五、遇到的挑戰與解決方式

### 1. 鄧白氏漲價談判破局，前期自動化投入歸零

鄧白氏單方面宣布漲價，談判過程中無法達成合理價格共識，導致先前為鄧白氏名單建置的自動化流程（每月清單、雙週名單自動化邏輯）全數失效，前期投入付諸流水。

**處置**：完成跨部門評估後，果斷做出終止合作決策，避免繼續投入資源；05-29 正式通知資訊部門，明確提出需與中華徵信所接洽資料接入方案，將後續系統對接風險轉移至資訊部門負責。

### 2. Claude Code 工作邊界管控難題

Claude Code 在執行任務時頻繁出現「過早宣告完成」的情形，需持續介入確認實際進度；同時在 VS Code 與 Cowork 分別開啟 Claude 實例，造成注意力分散與管理混亂。

**處置**：調整使用策略，採「監工模式」進行進度確認，並逐步收斂至單一工具操作，減少雙環境切換帶來的認知負擔；月末已摸索出較穩定的任務拆解與交辦方式。

### 3. 新企金系統資料一致性問題

進行三系統比對（新企金系統、原信系統、資料庫）時，發現新企金系統在不同日期查詢出的資料量不一致（前日大於後日），資料可信度存疑，無法作為可靠資料源。

**處置**：完成比對測試後決定不採用新系統查詢路徑，改回半年前規劃的直接串接方案，以穩定性為優先考量，避免後續資料錯誤風險。

### 4. 跨部門協調阻力

本月遭遇多起跨部門協調困難：對口經辦離職導致業務斷點（鄧白氏）、多次聯繫均未獲回應（法務、綠能部長）、提案通知漏發疑慮（排程小幫手機制不熟悉）。

**處置**：鄧白氏經辦離職透過電話確認完成接手；法務及綠能部長以信件持續追蹤；排程小幫手問題確認操作方式後完成補救，並記錄正確流程以備後續使用。

### 5. Token 消耗管理不當

05-20 使用 Opus 模型整頓 Schema 時，短時間內即耗盡當日 Token 配額，導致後續工作中斷。

**處置**：事後確認高耗能任務應預先評估 Token 用量，改以分批拆解方式處理大型任務，避免單次操作超量消耗。
